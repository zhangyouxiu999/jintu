# 点名页独立全平台（Web / iOS / Android / 小程序）方案

## 一、当前点名页能力与依赖梳理

### 1.1 功能清单

| 功能 | 说明 | 数据/接口 |
|------|------|-----------|
| 班级列表入口 | 首页展示班级，点击进入点名页 | 需班级列表 API（当前在 Next 服务端直查 DB） |
| 点名主界面 | 按班级 ID 展示学生列表、实时考勤状态 | `GET /api/classes/[code]` |
| 状态切换 | 未到(0) / 已到(1) / 请假(2) / 晚到(3) | Socket `attendance-update` + 可选持久化 |
| 拖拽排序 | 学生顺序调整并持久化 | `POST /api/classes/[code]` action: `update_order` |
| 搜索学生 | 按姓名过滤 | 纯前端 |
| 公告管理 | 发布/编辑/删除公告，今日/永久/自定义时间 | `POST /api/classes/[code]` 公告相关 + Socket `announcement-update` |
| 考勤报告 | 预览文本、复制、确认时批量写入数据库 | `POST` action: `batch_update_attendance` |
| 时段自动重置 | 上午/下午/晚一/晚二按本地时段清空状态 | 前端 localStorage + Socket 广播 |
| 全员到齐 / 重置考勤 | 批量标记、清空 | Socket `order-update`，不直接写库 |
| 日考勤矩阵 / 历史报表 | 跳转链接 | 依赖同站的 `/attendance/daily/[id]`、`/attendance/history/[id]` |

### 1.2 接口与实时通信

- **REST**
  - `GET /api/classes/[code]`：班级详情（学生、公告、考勤状态、排序）
  - `POST /api/classes/[code]`：`update_order` | `batch_update_attendance` | 公告 CRUD 等
- **Socket.io**
  - Path：`/api/socket_io_conn`（需先请求 `/api/socket_io` 拉起服务）
  - 事件：`join-room`、`attendance-update` → `attendance-sync`、`order-update` → `order-sync`、`announcement-update` → `announcement-sync`

### 1.3 前端技术栈（当前）

- **框架**：Next.js 15、React 19
- **UI**：Tailwind、Radix UI（Dialog、AlertDialog、Popover、Tooltip、Calendar 等）、shadcn 风格组件
- **交互**：@dnd-kit 拖拽、react-day-picker、date-fns、copy-to-clipboard、sonner、lucide-react
- **实时**：socket.io-client

### 1.4 后端依赖（供独立前端对接）

- MongoDB：Class、Student、Announcement、AttendanceLog、AttendanceHistory、Category
- Node + Socket.io 服务（可与现有 Next.js 同进程或拆成独立服务）

## 二、目标：点名页“单独做成全站本地项目”的两种理解

1. **独立前端仓库 + 可配置后端地址**  
   点名相关页面单独一个项目，可部署为 Web / 打包为 App / 小程序，所有数据仍通过现有（或独立部署的）后端 API + Socket 获取，前端“本地”指代码与部署独立，数据在云端。
2. **离线优先 / 纯本地**  
   无网或弱网时也能点名，数据先存本地，有网再同步（需后端增补同步 API 与冲突策略，工作量较大）。

下面方案按 **理解 1** 为主（独立前端 + 连接现有后端），并在第四节简要说明若要 **理解 2** 的扩展点。

## 三、全平台技术方案对比

### 3.1 方案 A：Taro / uni-app（推荐：Web + 小程序 + App 一套代码）

| 维度 | 说明 |
|------|------|
| 覆盖 | Web、微信/支付宝/抖音等小程序、iOS/Android App（基于 WebView 或原生壳） |
| 技术 | Vue 或 React（Taro 3 支持 React），一套代码多端编译 |
| 与现有点名页 | 需用 Taro/uni 组件重写页面，业务逻辑（状态、Socket、API 调用）可高度复用，UI 需适配各端 |
| 优点 | 小程序与 App 官方支持好，一套代码维护成本低 |
| 缺点 | 拖拽等复杂交互在小程序端需降级（如改为长按菜单调整顺序） |

**适用**：必须上架微信/支付宝等小程序，且希望 Web + App 共用一套逻辑时。

### 3.2 方案 B：Expo (React Native)（Web + iOS + Android，无小程序）

| 维度 | 说明 |
| 通道 | Web（Expo 实验性）、iOS、Android 原生应用 |
| 与现有点名页 | 用 React Native 组件重写 UI，API 与 Socket 逻辑可复用，需引入 socket.io-client（RN 兼容） |
| 优点 | 原生体验好、性能高，适合以 App 为主、Web 为辅 |
| 缺点 | 不直接产出小程序，若要小程序需再维护 Taro/uni 或原生小程序一版 |

**适用**：以 iOS/Android App 为主，小程序非刚需或可接受两套前端。

### 3.3 方案 C：现有 Next 前端 + Capacitor 打包 App + 小程序单独一版

| 维度 | 说明 |
| 通道 | Web：继续用 Next；App：用 Capacitor 把现有 Next 打包成 iOS/Android；小程序：用 Taro/uni 或原生重写一版 |
| 与现有点名页 | Web 与 App 共享同一套 Next 点名页代码；小程序单独实现，接口与 Socket 协议一致即可 |
| 优点 | Web/App 零重写，迭代快；小程序可按需精简功能 |
| 缺点 | 小程序需单独维护一套 UI 与路由，共三处代码（Next、Capacitor 配置、小程序） |

**适用**：希望 App 快速上线且与现有管理端完全一致，同时接受小程序单独一版。

### 3.4 方案 D：Flutter

| 维度 | 说明 |
| 通道 | Web、iOS、Android；小程序可通过第三方或官方能力（如微信 Flutter 小程序） |
| 与现有点名页 | 全部用 Dart 重写，仅复用接口契约与 Socket 事件设计 |
| 优点 | 性能与一致性较好，多端一致 |
| 缺点 | 重写工作量大，与现有 React 技术栈割裂 |

**适用**：技术栈选型以 Flutter 为主、可接受从零重写时。

## 四、推荐落地路线（按“独立前端项目 + 全平台”）

### 4.1 阶段一：点名页拆成独立 Web 项目（必做）

**目标**：点名相关功能单独一个前端仓库，不依赖 admin 其它页面，仅通过环境变量配置后端地址。

1. **新建仓库**（如 `jintu-attendance-web`）
   - 使用 **Vite + React + TypeScript**（或 Next 若需 SSR/API Route 复用），便于后续被 Taro/uni 或 Capacitor 复用逻辑。
2. **抽离并复用**
   - 从 admin 抽离：点名页主流程、学生列表、状态切换、公告、考勤报告、时段与重置逻辑。
   - 保留：Socket.io 连接（path 可配置）、`GET/POST /api/classes/[code]` 的封装。
   - UI：用 Vite 版 shadcn/ui 或自建一套基于 Radix 的组件，保证 Web 与后续 App 一致。
3. **路由与入口**
   - 班级列表：`GET /api/classes` 需在后端提供（当前在 Next 服务端直查 DB，可改为 `GET /api/classes` REST 接口）。
   - 点名页：`/attendance/:id`，仅依赖 `id`（班级 code）与上述 API。
   - 日考勤/历史报表：可做链接跳转回 admin 对应 URL，或在新项目中再做两页（需对应 API）。
4. **配置**
   - `VITE_API_BASE`、`VITE_SOCKET_PATH` 等，构建时注入，方便部署到不同环境（含本地）。

**交付物**：独立 Web 项目，部署后与现有点名行为一致，且不依赖 admin 其它路由。

### 4.2 阶段二：iOS / Android（在阶段一基础上二选一）

- **路径 2a：Capacitor 包一层**
  - 在独立 Web 项目中集成 Capacitor，打包成 iOS/Android App，同一套点名页代码，仅需处理深链、状态栏、安全区域等。
- **路径 2b：Taro/uni-app 重写 UI**
  - 用 Taro（React）或 uni-app 新开项目，复用阶段一中的 API 封装、Socket 事件与业务状态机，仅 UI 用 Taro/uni 组件重写；可同时产出 Web + 小程序 + App。

### 4.3 阶段三：小程序

- 若阶段二选的是 **Taro/uni**：同一项目即可编译出小程序。
- 若阶段二选的是 **Capacitor**：需用 Taro/uni 或微信原生再做一版“点名”小程序，接口与 Socket 与现有后端保持一致即可（可复用阶段一的接口文档与事件名）。

### 4.4 若要做“纯本地/离线优先”

- 前端：本地存储（IndexedDB / SQLite）保存班级快照、考勤状态、待同步队列。
- 后端：增加“同步”接口（按班级+时段上传/拉取差异、冲突策略如 last-write-wins 或服务端为准）。
- Socket：有网时连接，断线时用本地状态，恢复后按队列或全量同步。

## 五、接口与 Socket 契约（供独立项目对接）

独立前端只需依赖以下契约，与当前 admin 保持一致即可。

### 5.1 REST

- `GET /api/classes`  
  - 返回班级列表（当前若没有，需在后端新增，字段至少：code、name 等）。
- `GET /api/classes/:code`  
  - 返回：`id`(code)、`name`、`major_categories`、`students`（含 id、name、attendanceStatus）、`announcements`、`student_order`。
- `POST /api/classes/:code`  
  - Body 示例：
    - 排序：`{ "action": "update_order", "student_order": ["id1","id2",...] }`
    - 批量考勤：`{ "action": "batch_update_attendance", "student_ids": [...], "period": 0|1|2|3, "status_map": { "id": status } }`
    - 公告：`{ "content", "expiration_type", "custom_starts_at", "custom_expires_at" }` 或带 `announcement_id` 更新、`action: "delete", "announcement_id"` 删除。

### 5.2 Socket.io

- 连接 path：`/api/socket_io_conn`（或后端实际配置的 path）。
- 事件：
  - 客户端发：`join-room`(room)、`attendance-update`({ room, studentId, status })、`order-update`({ room, students })、`announcement-update`({ room, announcements })。
  - 服务端发：`attendance-sync`、`order-sync`、`announcement-sync`（payload 与现有 `pages/api/socket_io.js` 一致）。

## 六、小结

| 步骤 | 内容 |
|------|------|
| 1 | 梳理并固定点名页的 API + Socket 契约（见第五节） |
| 2 | 将点名页拆成独立 Web 项目（Vite + React），配置化后端地址，可选提供 `GET /api/classes` |
| 3 | 全平台选型：要小程序优先 → Taro/uni-app 重写 UI、复用逻辑；要 App 最快 → Capacitor 包现有 Web |
| 4 | 小程序：Taro/uni 同源产出，或 Capacitor 方案下单独做一版小程序并复用接口 |
| 5 | 若需离线/本地优先，再增加本地存储与同步 API 设计 |

当前项目中的点名页已具备完整的数据流与实时同步，独立成“全站本地项目”的核心是：**独立前端仓库 + 清晰的后端接口契约**，再按需选择 Taro/uni（Web+小程序+App）或 Next+Capacitor（Web+App）+ 小程序单独一版。
