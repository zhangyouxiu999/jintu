# 点名页 · 离线/纯本地设计方案（仅 iOS / Android）

> **产品定位**：轻量级考勤 App，**只需班级 + 学生姓名**，无大类、学号等额外字段。

## 一、目标与约束

| 项 | 说明 |
|----|------|
| 运行方式 | 完全离线，不连接任何后端与数据库 |
| 实时同步 | 不做，无 Socket |
| 平台 | **仅 iOS、Android 本地 App**（不做网页端、不做小程序） |
| 数据 | 全部存于本机（推荐原生 SQLite；或 WebView 内 IndexedDB） |
| 多设备 | 不保证一致，每台设备各自一份本地数据 |
| 分发 | 仅通过 App 安装包（商店 / 企业分发 / ad-hoc），无 Web 访问入口 |

---

## 二、功能范围

在「无网络、无服务端」前提下保留/调整的能力如下。

| 功能 | 在线版行为 | 离线/纯本地行为 |
|------|------------|------------------|
| 班级列表 | 接口拉取 | 从本地存储读取；**新增班级**（仅填班级名称）、**选择班级**（点击进入该班点名页）；支持编辑、删除班级 |
| 进入点名 | 接口拉取班级详情+学生+考勤 | 从本地存储读班级、学生；考勤状态从「当日当前时段」本地快照读 |
| 为班级添加学生 | - | **单独为某班级添加学生**，表单**仅需填写姓名**，id/排序由系统生成 |
| 状态切换 | 前端 + Socket 广播 + 可选写库 | 仅更新内存 + 立即写入本地考勤快照 |
| 拖拽排序 | 前端 + 写库 + Socket | 更新班级内学生顺序并写回本地班级数据 |
| 搜索学生 | 纯前端 | 不变 |
| 公告 | 接口 CRUD + Socket | 本地 CRUD，存班级关联的公告列表 |
| 考勤报告 | 预览 + 复制 + 确认时写库 | 预览 + 复制；「确认」仅把当前快照写入本地历史（用于日后查看），不连库 |
| 时段自动重置 | 按本地时段 + localStorage 标记 | 按本地时段；新时段对该班级+日期生成新快照或清空当前快照 |
| 全员到齐 / 重置考勤 | 前端 + Socket | 仅更新内存 + 写本地考勤快照 |
| 日考勤矩阵 / 历史报表 | 跳转其他页 + 接口 | 改为「本地历史」：按班级+日期+时段列出已保存的快照，可查看/导出 |

**班级与学生管理（必备）**：

- **新增班级**：用户可手动创建班级，**仅需填写班级名称**；创建后进入该班级或留在列表。
- **选择班级**：首页展示班级列表，点击某一班级即「选择该班级」，进入该班点名页或班级管理。
- **单独为班级添加学生**：在某个班级下可逐个添加学生，**只需填写姓名**即可；id、classId、sortIndex 由系统自动生成或分配。

**其他必备能力**：

- **数据导入**：支持从 JSON/Excel 导入「班级+学生」结构，用于初始化或从现有 admin 系统导出后倒入本机。
- **数据导出**：导出当前所有班级+学生+（可选）考勤历史，便于备份或迁移到其他设备。

## 三、本地数据模型

### 3.1 存储层级（仅 App）

- **推荐**：**原生 SQLite**（如 Capacitor 的 `@capacitor-community/sqlite`），数据存于 App 沙箱，与系统集成好、适合仅双端 App 的场景。
- **备选**：Capacitor 打包时 WebView 内使用 **IndexedDB**（无需原生插件，但与 Web 同源限制一致）。若先期追求实现速度可先用 IndexedDB，业务层通过统一 `store/*` 抽象，后续再切 SQLite。
- 业务层统一走「存储适配接口」（如 `classes.ts` / `attendance.ts` 等），不直接依赖具体实现，便于从 IndexedDB 迁到 SQLite。

### 3.2 实体与结构

```ts
// 班级（本地，轻量：仅名称）
interface ClassEntity {
  id: string;              // uuid
  name: string;            // 班级名称，用户唯一必填
  studentOrder: string[];   // 学生 id 顺序
  createdAt: string;
  updatedAt: string;
}

// 学生（本地，按班级归属；添加时仅需提供 name，其余由系统生成）
interface StudentEntity {
  id: string;              // 系统生成 uuid
  name: string;            // 用户唯一必填
  classId: string;
  sortIndex: number;       // 班内排序，新增时自动取当前最大+1
  createdAt: string;
  updatedAt: string;
}

// 考勤快照：某班级某日某时段的「状态」持久化
// 用于：进入点名页恢复状态、时段切换、报告确认后留档、历史查询
interface AttendanceSnapshot {
  id: string;
  classId: string;
  date: string;            // YYYY-MM-DD
  period: number;          // 0 上午 1 下午 2 晚一 3 晚二
  statusMap: Record<string, number>;  // studentId -> 0|1|2|3
  confirmedAt?: string;    // 报告「确认」时写入，用于区分未确认/已确认
  updatedAt: string;
}

// 公告（本地，按班级）
interface AnnouncementEntity {
  id: string;
  classId: string;
  content: string;
  expirationType: 'today' | 'permanent' | 'custom';
  startsAt?: string;
  expiresAt?: string;
  createdAt: string;
  updatedAt: string;
}
```

**可选简化**：若班级数量不多、每班人数有限，可将「班级 + 学生」合并为「班级」内嵌学生数组（如 `ClassEntity.students: { id, name, sortIndex }[]`），减少一次查询，写入时整班更新。

### 3.3 本地库设计（SQLite / IndexedDB 通用）

- **表/Store 名称**：`classes`、`students`、`attendance_snapshots`、`announcements`。
- **索引**：  
  - `students`: `classId`；  
  - `attendance_snapshots`: `(classId, date, period)`、可选 `(classId, date)`；  
  - `announcements`: `classId`。
- **主键**：各表 `id`（字符串 uuid）。
- **SQLite 补充**：`statusMap` 存为 JSON 文本列；若用 SQLite 插件，库文件放 App 文档/应用目录，按插件要求初始化即可。

## 四、核心流程

### 4.1 应用启动

1. 打开 App → 初始化本地库（SQLite 打开/建表，或 IndexedDB 打开/升级）。
2. 若为首次使用，展示「空状态」：引导新建班级或从文件导入数据。
3. 首页从 `classes` 读取列表（按 `updatedAt` 排序），展示班级入口。

### 4.2 进入点名页

1. 路由参数：班级 `id`。
2. 从本地读：`ClassEntity`、该班的 `StudentEntity` 列表（按 `studentOrder` 或 `sortIndex` 排序）、该班的 `AnnouncementEntity`（按生效时间过滤）。
3. 当前日期、当前时段（0~3）由前端根据本地时间计算。
4. 用 `(classId, date, period)` 查 `AttendanceSnapshot`：
   - 有则用 `statusMap` 恢复每个学生的 `attendanceStatus`；
   - 无则全部为 0（未到）。
5. 界面渲染：学生列表、公告、底部统计、报告入口均只依赖上述本地数据。

### 4.3 状态切换（未到/已到/请假/晚到）

1. 更新内存中的学生状态。
2. 立即写回本地：对该 `(classId, date, period)` 做 upsert 更新 `statusMap` 和 `updatedAt`（不设 `confirmedAt`）。
3. 无网络、无 Socket。

### 4.4 拖拽排序

1. 更新内存中的学生顺序。
2. 更新 `ClassEntity.studentOrder`（或各 `StudentEntity.sortIndex`）并写回 `classes`（及必要时 `students`）。
3. 无网络。

### 4.5 公告 CRUD

1. 增删改均写 `announcements` 表，带 `classId`。
2. 列表展示时按 `startsAt/expiresAt` 与当前时间过滤。

### 4.6 考勤报告

1. **预览**：由当前内存中的学生状态生成文案；轻量版**仅含班级名、日期时段、应到/实到/请假/晚到/未到名单**，不含大类、教学老师到岗等。
2. **复制 / 分享**：写入剪贴板；移动端可同时提供「分享」到其他 App（如微信、邮件），使用 Capacitor Share API。
3. **确认**：
   - 将当前 `(classId, date, period)` 的 `AttendanceSnapshot` 的 `confirmedAt` 设为当前时间（若之前是未确认快照则更新，否则同一条已存在则只更新 `confirmedAt`）；
   - 不连接数据库，仅本地标记「已确认」，便于历史里区分「已提交报告」的记录。

### 4.7 时段自动重置

1. 根据本地时间计算当前时段（如 0 上午、1 下午、2 晚一、3 晚二）。
2. 进入点名页或定时检测：若发现「当前时段」相对上次已变化，则：
   - 当前内存状态视为「新时段」：全部置为未到；
   - 对新时段做一次 `AttendanceSnapshot` 的 upsert（statusMap 全 0），或仅清空内存并在下次离开/保存时再写（按产品偏好二选一）。
3. 不依赖服务端与 Socket。

### 4.8 全员到齐 / 重置考勤

1. 仅更新内存 + 对应当前 `(classId, date, period)` 写回 `AttendanceSnapshot`（全员到齐：statusMap 全 1；重置：全 0）。
2. 无网络。

### 4.9 本地历史（替代「日考勤矩阵 / 历史报表」）

1. **列表**：按班级筛选，按日期、时段展示 `AttendanceSnapshot`（建议只展示 `confirmedAt` 有值的，或同时支持「未确认」）。
2. **详情**：某条快照的 statusMap 展开为名单（未到/已到/请假/晚到）。
3. **导出**：将历史导出为 JSON/CSV 文件后，通过系统分享或保存到本地/云盘（Capacitor Share、Filesystem 或文件选择器）。

## 五、数据导入与导出

### 5.1 导入（初始化/从 admin 迁移）

- **格式**：JSON 为首选（结构清晰、易从现有 admin 生成）；可选支持 Excel（解析为「班级名、学生名单」等）。
- **最小结构示例**：
  ```json
  {
    "classes": [
      { "id": "class-uuid-1", "name": "某某班" }
    ],
    "students": [
      { "id": "stu-1", "name": "张三", "classId": "class-uuid-1", "sortIndex": 0 }
    ]
  }
  ```
- **行为**：与现有本地数据「合并」或「覆盖」由产品决定（建议首次导入为覆盖，之后可选合并并生成新 id 避免冲突）。
- **入口**：设置页或首页「导入」按钮，通过系统文件选择器（Capacitor 或原生 File Picker）选择 JSON/Excel 文件，解析后写入本地库。

### 5.2 导出（备份/迁移）

- **内容**：当前所有 `classes`、`students`；可选 `attendance_snapshots`、`announcements`。
- **格式**：JSON（便于再导入或备份）。
- **入口**：设置页或首页「导出」按钮，生成文件后通过 **分享**（Share API）或 **保存到设备/云盘**（Filesystem + 系统保存面板）交给用户，无「浏览器下载」。

## 六、技术栈建议（仅 iOS / Android）

| 层次 | 建议 | 说明 |
|------|------|------|
| 框架 | Vite + React + TypeScript（或 Expo 若选 RN） | 与现有点名页逻辑接近；**仅构建为 App**，不部署 Web |
| 打包 | **Capacitor** | 用 Web 技术栈打包成 iOS/Android，一套代码双端；无 Web 发布入口 |
| 本地存储 | **SQLite**（推荐）或 IndexedDB | SQLite：`@capacitor-community/sqlite`，数据在 App 沙箱；IndexedDB：WebView 内可用，无需原生插件 |
| UI | Tailwind + Radix/shadcn（或 React Native 组件若用 RN） | 点名页组件可复用；需适配安全区、状态栏、全面屏 |
| 路由 | React Router（Hash 或 Memory 路由） | App 内：班级列表、点名页、设置（导入/导出）、本地历史；**不用 BrowserRouter**（无 URL 栏） |
| 原生能力 | Capacitor：Share、Filesystem、可选 File Picker | 导入：选文件；导出/报告：分享或保存到设备 |
| 构建产物 | 仅 **iOS 包**（.ipa / Xcode）与 **Android 包**（.aab/.apk） | 不产出 Web 构建或 PWA |

**移除/不涉及**：`socket.io-client`、所有 `fetch('/api/...')`、Next.js、PWA、Web 部署；不再依赖任何后端与数据库。

**仅双端时的可选替代**：若希望更偏原生体验，可选用 **Expo (React Native)** 或 **Flutter**，则 UI 与导航按各自生态实现，数据层仍为本地 SQLite/AsyncStorage 等，业务逻辑（时段、报告文案、快照模型）可复用。

### 6.1 仅 App 时的注意点

| 项 | 说明 |
|----|------|
| 安全区 / 状态栏 | 使用 `env(safe-area-inset-*)` 或 Capacitor StatusBar 插件，避免刘海、底部横条遮挡 |
| 返回 | 无浏览器返回键；列表→点名页需导航栏「返回」或系统返回手势（Capacitor 内多为 WebView 自带） |
| 导入/导出 | 通过系统文件选择器、分享面板、保存到文件，不依赖浏览器 `download` / `input[file]` |
| 复制/分享报告 | 剪贴板 + Share API，便于发到微信等 |
| 无 Web 部署 | 不配置域名、不部署静态资源；仅通过应用商店或企业分发安装 |

## 七、目录与模块划分（建议）

```
src/
  store/           # 本地持久化
    db.ts           # IndexedDB 初始化、库名与版本
    classes.ts      # 班级 CRUD
    students.ts     # 学生 CRUD
    attendance.ts   # AttendanceSnapshot 读写、当前时段计算
    announcements.ts
  hooks/
    useClass.ts
    useAttendance.ts
    useAnnouncements.ts
  pages/
    ClassList.tsx
    Attendance.tsx   # 点名主页（从现有 admin 迁入并改为读 store）
    History.tsx      # 本地历史列表/详情
    Settings.tsx     # 导入/导出
  components/
    ...
  lib/
    period.ts       # 时段 0~3 计算
    reportText.ts   # 报告文案生成
  types/
    ...
```

业务层只依赖 `store/*` 与 `lib/*`，不出现 `fetch`/Socket，便于测试与后续换存储实现。

## 八、与现有点名页的差异小结

| 项目 | 现有（在线 Web） | 离线/纯本地（仅 iOS+Android） |
|------|------------------|-------------------------------|
| 平台 | 浏览器 | 仅 App 安装包，无网页端 |
| 班级/学生来源 | 接口 | 本地 SQLite/IndexedDB + 文件导入 |
| 考勤状态 | 接口 + Socket | 本地 AttendanceSnapshot |
| 公告 | 接口 + Socket | 本地 AnnouncementEntity |
| 报告确认 | 写库 | 仅写本地快照 + confirmedAt |
| 报告分发 | 复制 | 复制 + 系统分享（如微信、邮件） |
| 多端一致 | Socket 同步 | 不保证，每端独立 |
| 导入/导出 | - | 文件选择器 + 分享/保存到设备 |
| 新增 | - | 导入、导出、本地历史页、仅双端构建 |

按此设计即可实现「仅 iOS/Android、离线/纯本地、无 Socket、不连数据库」的点名 App；若后续要加「可选同步」，可在本方案基础上增加同步模块与冲突策略，而不改动核心本地模型与流程。
