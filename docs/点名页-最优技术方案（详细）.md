# 点名页 · 最优技术方案（详细）

> **目标**：仅 iOS / Android 本地 App，离线纯本地、无后端、无 Socket、无网页端。  
> **产品定位**：轻量级考勤 App，**只需班级名称 + 学生姓名**，无大类、学号等字段。

---

## 一、方案选型结论

### 1.1 最优方案：Capacitor + Vite + React + TypeScript + SQLite

| 维度 | 选定方案 | 理由 |
|------|----------|------|
| **应用壳** | **Capacitor** | 一套 Web 技术栈打包成 iOS/Android，官方维护、生态成熟；仅构建 App，不部署 Web。 |
| **前端框架** | **Vite + React + TypeScript** | 与现有点名页（admin 项目）同技术栈，**可直接迁移页面与逻辑**，仅替换数据源为本地存储；Vite 构建快、无 Next 依赖。 |
| **本地存储** | **SQLite**（`@capacitor-community/sqlite`） | 数据落 App 沙箱，与系统一致；适合仅双端、无 Web 的场景；避免 WebView 内 IndexedDB 的兼容与容量顾虑。 |
| **UI** | **Tailwind + Radix UI / shadcn 风格** | 与现有点名页一致，迁移时样式与交互可复用；通过 CSS 安全区变量适配刘海与底部横条。 |
| **路由** | **React Router（HashRouter）** | App 内无地址栏，Hash 路由即可；便于从现有 SPA 迁入。 |
| **原生能力** | **Capacitor 插件**：Share、Filesystem、StatusBar、可选 File Picker | 报告分享、导入/导出文件、状态栏与安全区。 |

**核心优势**：在「仅双端、离线本地」前提下，**最大化复用现有 admin 点名页的 React 组件与业务逻辑**，只做「去掉网络与 Socket、接入本地 SQLite」的改造，开发与维护成本最低。

## 二、备选方案对比（为何不选）

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Expo (React Native)** | 原生 UI、滚动与性能更好、生态成熟 | 需用 RN 组件重写整页 UI，无法直接复用现有 DOM 组件；业务逻辑可复用 | 若优先追求「纯原生观感」且接受一次性 UI 重写 |
| **Flutter** | 性能与一致性最佳、单代码库双端 | 需用 Dart 全量重写，与现有 React 代码零复用 | 团队以 Flutter 为主或从零立项时 |
| **Capacitor + IndexedDB** | 无需原生 SQLite 插件，实现更快 | WebView 内 IndexedDB 与真机/系统备份行为不完全一致；数据量大时不如 SQLite 可控 | 仅作短期验证或数据量极小时 |

在「现有代码为 React、希望尽快交付仅双端 App」的前提下，**Capacitor + Vite + React + SQLite** 为当前最优。

## 四、项目与目录结构

### 4.1 仓库与构建关系

- **单仓库**：一个前端工程（如 `jintu-attendance-app`）。
- **构建**：`npm run build` 产出 `dist/`，由 Capacitor 将 `dist` 作为 Web 资源打包进 iOS/Android 工程；**不部署 dist 到任何 Web 服务器**。
- **双端**：`npx cap add ios` / `npx cap add android` 后，通过 Xcode 与 Android Studio 分别打出 .ipa 与 .aab/.apk。

## 五、存储层详细设计（SQLite）

### 5.1 库与连接

- **库名**：`jintu_attendance.db`。
- **位置**：由 `@capacitor-community/sqlite` 管理，通常位于 App 文档/应用目录（插件文档指定）。
- **初始化**：App 启动时（如 `main.tsx` 或首屏前）调用一次 `db.init()`：若库不存在则创建并执行建表；若已存在则仅打开连接。所有读写均通过异步接口，避免阻塞 UI。

### 5.2 表结构（SQL）

```sql
-- 班级（轻量：仅名称）
CREATE TABLE IF NOT EXISTS classes (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  student_order TEXT NOT NULL,  -- JSON 数组字符串，如 '["id1","id2"]'
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

-- 学生
CREATE TABLE IF NOT EXISTS students (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  class_id TEXT NOT NULL,
  sort_index INTEGER NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (class_id) REFERENCES classes(id)
);
CREATE INDEX IF NOT EXISTS idx_students_class_id ON students(class_id);

-- 考勤快照
CREATE TABLE IF NOT EXISTS attendance_snapshots (
  id TEXT PRIMARY KEY,
  class_id TEXT NOT NULL,
  date TEXT NOT NULL,           -- 'YYYY-MM-DD'
  period INTEGER NOT NULL,     -- 0|1|2|3
  status_map TEXT NOT NULL,    -- JSON 对象字符串，如 '{"sid1":1,"sid2":0}'
  confirmed_at TEXT,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (class_id) REFERENCES classes(id),
  UNIQUE(class_id, date, period)
);
CREATE INDEX IF NOT EXISTS idx_attendance_class_date_period ON attendance_snapshots(class_id, date, period);
CREATE INDEX IF NOT EXISTS idx_attendance_class_date ON attendance_snapshots(class_id, date);

-- 公告
CREATE TABLE IF NOT EXISTS announcements (
  id TEXT PRIMARY KEY,
  class_id TEXT NOT NULL,
  content TEXT NOT NULL,
  expiration_type TEXT NOT NULL,  -- 'today'|'permanent'|'custom'
  starts_at TEXT,
  expires_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (class_id) REFERENCES classes(id)
);
CREATE INDEX IF NOT EXISTS idx_announcements_class_id ON announcements(class_id);
```

### 5.3 存储层 API 约定（TypeScript）

以下为业务层调用的抽象接口，内部由 `db.ts` 使用 SQLite 插件执行 SQL。

- **classes**: `getAll()`, `getById(id)`, `insert(entity)`, `update(entity)`, `delete(id)`；`update` 时至少更新 `student_order` 与 `updated_at`。
- **students**: `getByClassId(classId)`, `insert(entity)`, `update(entity)`, `delete(id)`, `replaceListForClass(classId, students[])`（用于导入时整班替换）。添加学生时业务层只需传入 `{ name, classId }`，由 store 生成 id、sortIndex（如取该班当前 max(sortIndex)+1）并写入。
- **attendance_snapshots**: `get(classId, date, period)`, `upsert(snapshot)`, `listByClassAndDate(classId, date?)`, `listConfirmedByClass(classId)`（历史页用）。
- **announcements**: `getByClassId(classId)`, `insert(entity)`, `update(entity)`, `delete(id)`；业务层按 `starts_at`/`expires_at` 过滤生效中的公告。

所有时间字段统一 **ISO 8601 字符串**（如 `new Date().toISOString()`）；`status_map`、`student_order` 存取时 `JSON.stringify` / `JSON.parse`。

## 六、路由与页面

### 6.1 路由表

| 路径 | 组件 | 说明 |
|------|------|------|
| `/` | ClassList | 班级列表（首页） |
| `/attendance/:classId` | Attendance | 点名主界面 |
| `/history/:classId?` | History | 本地历史，可选班级筛选 |
| `/settings` | Settings | 导入、导出、关于等 |

使用 **HashRouter**，实际路径为 `#/`、`#/attendance/xxx` 等，避免与原生壳或未来深度链接冲突。

### 6.2 页面职责简述

- **ClassList**：从 `store.classes` 读列表；**新增班级**入口（新建班级，**仅填班级名称**）；**选择班级**：点击某一项即进入该班点名页 `#/attendance/:id`；入口「导入」「设置」。
- **Attendance**：从 `store` 读班级、学生、公告、当前时段快照；**单独为班级添加学生**：提供「添加学生」入口，弹窗或页内表单**仅需填写姓名**，id/classId/sortIndex 由系统生成；状态变更与排序写回 `store`；报告预览/复制/分享/确认；底部统计与公告入口；导航栏返回、更多（全员到齐、重置、历史入口）。
- **History**：从 `store.attendance_snapshots` 按班级/日期查；列表展示确认过的快照，详情展示名单；支持导出为 JSON/CSV 并通过 Share 或保存到文件。
- **Settings**：导入（选 JSON/Excel 文件 → 解析 → 写入 classes+students）、导出（生成 JSON → Share 或保存）、可选「关于」。

### 6.3 班级与学生管理功能（必做）

| 功能 | 说明 | 实现要点 |
|------|------|----------|
| **新增班级** | 用户手动创建班级 | 弹窗或独立页：**仅填班级名称**；提交后写入 `store.classes`，可跳转该班或留在列表。 |
| **选择班级** | 从列表进入某个班级 | 班级列表每项可点击，进入 `#/attendance/:classId` 点名页（即「选择该班级」）。 |
| **单独为班级添加学生** | 在某个班级下逐个添加学生 | 在点名页或班级管理入口提供「添加学生」；表单**只需填写姓名**，提交时由前端生成 uuid 作为 id、当前 classId、sortIndex 取该班当前最大+1，写入 `store.students` 并更新 `classes.student_order`。 |

添加学生时**仅需姓名**，无需学号、手机号等；若需编辑/删除学生，可在学生列表项上提供长按或更多菜单。

## 七、原生能力与适配

### 7.1 分享（报告 / 导出文件）

- **插件**：`@capacitor/share`。
- **用法**：报告文案或导出文件（先写临时路径）通过 `Share.share({ title, text, url, dialogTitle })` 调起系统分享面板；可分享到微信、邮件、保存到文件等。

### 7.2 文件导入

- **方式一**：`@capacitor/filesystem` 不直接提供「用户选文件」；通过 **Capacitor 的 File Picker 社区插件** 或 **原生桥接** 让用户从系统选择文件，拿到 URI 或 base64 后由前端解析 JSON/Excel 并写入 store。
- **方式二**：若暂时不做「选文件」，可做「从剪贴板粘贴 JSON」作为简易导入，或通过分享进来的文件（如从微信「用其他应用打开」）在 App 内接收。

### 7.3 文件导出与保存

- 将导出内容生成 JSON 字符串，用 `Filesystem.writeFile` 写入临时目录（Capacitor 提供的 cache 或 documents 路径），得到 `filePath` 后传给 `Share.share({ url: filePath })`，用户选择「保存到文件」或分享到其他 App。
- 若需「保存到相册/指定文件夹」：iOS/Android 需用 Filesystem 与系统保存面板或原生插件配合，可选后续迭代。

### 7.4 状态栏与安全区

- **StatusBar**：`@capacitor/status-bar` 设置样式（深色/浅色）、背景色，与 App 主题一致。
- **安全区**：在根布局或主内容区使用 `padding: env(safe-area-inset-top) env(safe-area-inset-right) env(safe-area-inset-bottom) env(safe-area-inset-left)`；或通过 Tailwind 的 `pb-safe` 等工具类包裹底部栏，避免刘海与 Home 条遮挡。

### 7.5 返回与导航

- 无浏览器返回键；列表到点名页使用导航栏「返回」按钮（或图标），路由使用 `useNavigate(-1)` 或 `navigate('/')`。
- 若需支持系统返回手势（如 iOS 左滑），Capacitor WebView 内多数情况下会与前端路由联动；必要时可在原生层监听返回并通知前端。

## 八、从现有 admin 点名页迁移要点

### 8.1 可直接复用

- **业务逻辑**：时段计算（`getCurrentPeriodId`）、报告文案模板（应到/实到/请假/晚到/未到）、学生状态枚举（0/1/2/3）。
- **组件结构**：StudentAttendanceList、SortableStudentRow、AttendanceStats、AnnouncementManager、考勤报告 Dialog、搜索框与筛选。
- **类型定义**：Student、班级名、AnnouncementItem 与 admin 的 `types` 对齐后迁入 `src/types`（轻量版不包含大类字段）。
- **UI 与样式**：Tailwind 类名、Radix 组件用法、拖拽 @dnd-kit 配置，在无 DOM 差异的前提下可直接拷贝。

### 8.2 必须移除或替换

- **移除**：所有 `fetch('/api/...')`、`io(socket...)`、与 `next/navigation`（改用 `react-router-dom` 的 `useParams`、`useNavigate`）。
- **替换**：  
  - 班级/学生/公告的「拉取」改为从 `store.classes`、`store.students`、`store.announcements` 读；  
  - 状态变更、排序、公告 CRUD、报告确认 改为写 `store.attendance` / `store.classes` / `store.announcements`；  
  - 报告「复制」保留；增加「分享」按钮，调用 `Share.share({ text: reportText })`。

### 8.3 需新增

- **班级列表页**：admin 为服务端渲染的首页，App 内改为从 store 读班级列表的客户端页面；**新增班级**入口（新建班级，**仅填班级名称**）；**选择班级**即点击列表项进入 `#/attendance/:classId`；并增加「导入」「设置」入口。
- **新增班级**：弹窗或简单页，**仅填班级名称**，提交后写 `store.classes`。
- **单独为班级添加学生**：在点名页（或班级管理）提供「添加学生」，表单**仅需填写姓名**，id/classId/sortIndex 由系统生成并写 `store.students` 与更新 `classes.student_order`。
- **设置页**：导入（文件或剪贴板）、导出、关于。
- **本地历史页**：按班级/日期/时段展示 `attendance_snapshots`，详情与导出。

## 九、导入 / 导出格式（与 admin 互通）

### 9.1 导出 JSON 结构（可由 admin 生成供 App 导入）

```json
{
  "version": 1,
  "exportedAt": "2025-02-28T12:00:00.000Z",
  "classes": [
    {
      "id": "uuid-1",
      "name": "某某班",
      "studentOrder": ["sid1", "sid2"]
    }
  ],
  "students": [
    { "id": "sid1", "name": "张三", "classId": "uuid-1", "sortIndex": 0 },
    { "id": "sid2", "name": "李四", "classId": "uuid-1", "sortIndex": 1 }
  ]
}
```

可选增加 `attendance_snapshots`、`announcements` 用于全量备份；导入时可按「覆盖」或「合并」策略（建议首次为覆盖，合并时为新班级/学生生成新 id 避免冲突）。

### 9.2 历史导出（仅快照）

- 按班级、日期范围筛选后，导出为 JSON 或 CSV；结构可为 `{ classId, date, period, statusMap, confirmedAt }[]` 或扁平化名单行，便于在 Excel 中查看。

## 十、构建与分发

### 10.1 开发

- `npm run dev`：Vite 开发服务器；浏览器中调试界面与路由（存储需 mock 或临时使用 IndexedDB/localStorage，或仅在真机/模拟器用 SQLite）。
- `npx cap run ios` / `npx cap run android`：在模拟器或真机中运行，使用真实 SQLite 与原生能力。

### 10.2 生产构建

1. `npm run build` → 生成 `dist/`。
2. `npx cap sync` → 将 `dist` 拷贝到 `ios/App/App/public` 与 `android` 对应资源目录，并更新原生依赖。
3. **iOS**：用 Xcode 打开 `ios/App/App.xcworkspace`，选择目标设备与签名，Archive 后导出 .ipa（商店或 ad-hoc）。
4. **Android**：用 Android Studio 打开 `android`，Build Bundle(s) / APK(s)，生成 .aab 或 .apk。

### 10.3 不产出

- 不部署 `dist` 到任何 Web 服务器；不配置 PWA 或 Service Worker；不提供通过 URL 访问的网页版。

## 十一、风险与应对

| 风险 | 应对 |
|------|------|
| SQLite 插件在部分机型/系统上的兼容性 | 选用 `@capacitor-community/sqlite` 较新版本并跟随 Capacitor 主版本；上线前在目标最低系统版本上做真机测试。 |
| 大名单（如数百人）下滚动与拖拽性能 | 列表虚拟化（如 react-window）；拖拽在移动端可保留或降级为「长按调序」以减少手势冲突。 |
| 用户不会「选文件」导致导入困难 | 提供「从剪贴板粘贴 JSON」；或由 admin 提供「生成导入包」并通过微信等发给自己，App 内通过「用其他应用打开」接收。 |
| 双端 UI 差异（如 Android 返回键） | 统一在导航栏提供「返回」；必要时在 Android 上监听硬件返回并映射到路由返回。 |

---

## 十二、小结

- **最优技术方案**：**Capacitor + Vite + React + TypeScript + SQLite**，仅构建 iOS 与 Android App，无网页端、无后端、无 Socket。
- **存储**：SQLite 四张表（classes、students、attendance_snapshots、announcements），业务层通过 `store/*` 抽象访问。
- **原生能力**：Share、Filesystem、StatusBar、可选 File Picker，用于报告分享、导入/导出与安全区适配。
- **迁移**：从 admin 点名页复用组件与业务逻辑，去掉网络与 Socket，接入本地 store；新增班级列表、设置、本地历史与导入/导出。
- **构建与分发**：仅产出 iOS 与 Android 安装包，通过应用商店或企业/ad-hoc 分发。

按此详细方案可实现「仅 iOS/Android、离线纯本地」的点名 App，并与现有 admin 在数据格式上互通（导入/导出 JSON）。
