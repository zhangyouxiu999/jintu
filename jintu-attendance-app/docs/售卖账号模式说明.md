# 售卖账号方式卖本地 App

商业模式：**卖的是「账号」**，客户用账号在**本地 App** 里登录后使用，数据仍存在本机，不依赖云端同步。

---

## 两种实现方式

### 方式一：一客户一包（无需后端）

- **做法**：每卖出一个账号，就为该客户打一版**专属安装包**，包内写入该客户唯一的账号密码（通过构建时 `VITE_LOGIN_USERNAME` / `VITE_LOGIN_PASSWORD`）。
- **交付**：把安装包（APK/IPA 或下载链接）发给客户，并告知「打开 App 即已登录」或提供同一组账号密码让其自行输入（若包内已写死则无需输入）。
- **优点**：无需任何服务器，当前项目已支持（生产构建必配登录凭据即可）。  
- **缺点**：一个包对应一个账号，客户多时需自动化打包（如 CI 按订单生成不同 .env 再构建）；不适合在应用商店「一个通用 App」上架后让用户自行购买账号。

**适用**：客户量可控、以定制/内购形式交付（如按学校单独售卖）。

---

### 方式二：一个通用包 + 多账号（需简易登录校验）

- **做法**：只维护**一个** App 安装包（可上架应用商店或官网下载）。用户购买后获得「账号+密码」，在 App 登录页输入。App 在登录时请求**你的校验接口**（仅做账号密码校验，不存业务数据），校验通过后允许进入，之后全部在本地使用，数据仍存本机。
- **后端**：只需一个**最小服务**，例如：
  - 接口：`POST /api/auth/login`，入参 `username`、`password`，在数据库中查用户表，校验通过返回 200 + 简单 token（或仅 200），失败返回 401。
  - 数据库：一张「账号表」即可（用户名、密码哈希、是否有效、到期时间等）。
- **前端**：登录时先调该接口，成功后再执行当前的「保存登录状态、跳转首页」逻辑；不成功则提示账号或密码错误。
- **优点**：一个包服务所有客户，适合应用商店或官网统一分发；可随时在后台增删账号、设有效期。  
- **缺点**：需要部署并维护一个小的认证服务（可极简：单机 + SQLite/MySQL + 一个登录接口即可）。

**适用**：希望「一个 App + 卖很多账号」、或上架商店后用户凭购买得到的账号登录。

---

## 对比小结

| 项目         | 方式一：一客户一包     | 方式二：通用包 + 多账号   |
|--------------|------------------------|----------------------------|
| 是否需要后端 | 否                     | 是（仅登录校验）          |
| 安装包数量   | 每客户一包（可自动化） | 一个通用包                |
| 账号管理     | 构建时写进包或人工发   | 后台数据库，可增删/过期   |
| 应用商店     | 不适合「一个 App 卖账号」 | 适合                     |
| 当前代码     | 已支持                 | 需增加「登录时调接口校验」 |

---

## 若选方式一（一客户一包）

1. **销售流程**：客户付款 → 你为其生成唯一用户名/密码（或客户自选）→ 用该凭据打一版包（或把凭据写进交付包对应的 .env 再构建）。
2. **自动化建议**：用脚本或 CI（如 GitHub Actions）根据订单或名单，循环：生成 `.env`（含该客户 `VITE_LOGIN_*`）→ `npm run build` → `cap sync` → 打 APK/IPA，产出按客户编号或名称命名的安装包。
3. **安全**：不要把多客户的凭据放在同一个仓库或同一份 .env 里；构建产物按客户单独存放或分发。

---

## 若选方式二（通用包 + 多账号）

1. **后端**：实现 `POST /api/auth/login`，查账号表校验用户名与密码（密码需哈希存储），返回 200 + token（可选）或仅 200。
2. **前端**：  
   - 登录页提交时先请求该接口；  
   - 成功：再执行现有「保存登录状态、跳转首页」；  
   - 失败：提示「账号或密码错误」或「账号已失效」。  
3. **账号表**：至少包含 用户名、密码哈希、状态（启用/禁用）、可选到期时间；新售出账号时在后台插入一条记录即可。

这样即可实现「售卖账号 + 本地 App」：**卖的是账号，用的是本地 App，校验只做登录那一步。**

## 防止 APK 转发导致一号多用

**问题**：客户把 APK 转发给别人，或把账号密码告诉别人，就会出现「一个购买、多人使用」。

**核心做法：登录账号与设备标识绑定**  
即：后端为每个账号维护「已绑定的设备标识列表」（如每账号最多 1～2 条）。用户在某台设备上登录时，App 把当前设备的标识（如 Android ID、或由多信息生成的设备指纹）和账号一起发给后端；后端校验账号密码通过后，再判断该设备是否在绑定列表中：在则允许登录并可选刷新绑定时间；不在且未超限则加入列表并允许登录；不在且已超限则拒绝或要求先解绑一台旧设备。这样，同一账号只能在已绑定的那几台设备上用，转发 APK 或泄露密码给他人，对方在新设备上登录会被拒绝（或顶掉原用户，由你策略决定）。

- **方式一（一客户一包）**：凭据写在包内，谁拿到 APK 谁就能用，**无法从技术上限制**转发。只能靠合同/授权条款约束，或仅面向可信客户（如学校签协议）。
- **方式二（通用包 + 登录校验）**：可以在**校验接口**上做限制，降低一号多用的可能：

| 手段 | 做法 | 效果 |
|------|------|------|
| **设备绑定** | 登录时把设备标识（如 Android ID / 设备指纹）发给后端；后端规定每账号最多绑定 1～2 台设备，新设备登录时拒绝或要求「踢掉一台旧设备」后再登。 | 即使别人拿到账号密码和 APK，在新设备上登录会被拒或顶掉原用户，显著增加转发成本。 |
| **定期联网校验** | App 在启动时（或每隔 N 天）请求一次后端，例如 `GET /api/auth/check` 带 token，后端校验账号是否有效、是否超设备数、是否过期。若失败则**强制登出**，回到登录页。 | 纯离线长期用会失效，必须偶尔联网；转发出去的人一旦被踢或账号过期，就无法继续用。 |
| **并发/在线限制** | 可选：同一账号只允许一台设备「在线」，后登录的踢掉先登录的（需后端记录当前 token/设备）。 | 限制同一时刻只有一人使用，但对「错峰使用」仍可能一号多用。 |

**建议**：若担心转发，优先用**方式二**，并在后端实现「设备绑定 + 启动时（或定期）联网校验」；业务数据仍可全部在本地，只是登录与校验依赖你的服务。纯方式一、无后端时，无法从技术上杜绝 APK 转发，只能靠条款与商务约束。
