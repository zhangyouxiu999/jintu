# 商用化路线图

将「稍微一点」考勤 App 从内网/演示用途升级为可商用产品。

**推荐商业模式：售卖账号 + 本地 App**  
即：卖的是「账号」，客户用账号在本地 App 里登录使用，数据仍存本机，无需云端同步。  
两种实现方式（一客户一包 vs 通用包+多账号）及选型见 **[售卖账号模式说明.md](./售卖账号模式说明.md)**。

---

## 一、阶段概览

| 阶段 | 目标 | 优先级 |
|------|------|--------|
| **1. 安全与配置** | 去掉默认账号、可配置品牌、环境区分 | 必须（已完成） |
| **2. 售卖账号** | 按「售卖账号模式说明」选方式一或二并落地 | 必须 |
| **3. 合规与运维** | 隐私政策、应用商店上架、更新策略 | 建议 |
| **4. 数据上云（可选）** | 若日后需多端同步/备份，再接入 API + 数据库 | 按需 |

---

## 二、阶段 1：安全与配置（当前仓库即可完成）

- **生产环境禁用默认账号**  
  - 生产构建下不再使用 `admin/admin`，必须配置 `VITE_LOGIN_USERNAME` 与 `VITE_LOGIN_PASSWORD`，否则登录页提示「未配置登录凭据」。
- **品牌可配置**  
  - 应用名称、登录页标题等通过 `VITE_APP_NAME` 等环境变量注入，便于白标或换品牌。
- **文档**  
  - `.env.example` 与 README 中说明各变量含义、生产环境必填项及安全注意点。

**交付物**：代码改造 + `.env.example` 与文档更新。

---

## 三、阶段 2：售卖账号落地

完全按 **[售卖账号模式说明.md](./售卖账号模式说明.md)** 执行即可：

- **方式一（一客户一包）**：无需后端；为每个客户用其账号打专属包，或交付时提供其专属账号。当前代码已支持，只需在构建时配置该客户的 `VITE_LOGIN_USERNAME` / `VITE_LOGIN_PASSWORD`，并可脚本化/CI 批量打包。
- **方式二（通用包 + 多账号）**：需做一个**仅做登录校验**的后端（例如 `POST /api/auth/login` + 账号表）；前端登录时先请求该接口，通过后再按现有逻辑进入本地使用。数据仍全部在本地，后端不存业务数据。

---

## 四、阶段 3：合规与运维

- **隐私与条款**  
  - 隐私政策、用户协议；若收集学生/教师信息，需符合当地法规（如告知、同意、最小必要）。
- **日志与监控**  
  - 后端记录操作日志、登录日志；可选错误上报与性能监控。
- **发布与更新**  
  - 应用商店上架规范（包名、签名、版本号）；Capacitor 应用可配合热更新或强制升级策略。

---

## 五、阶段 4：数据上云（可选）

若日后希望支持**多端同步、云端备份或统一管理**，再考虑：

- 后端提供班级/学生/考勤等 API + 数据库，前端在保留本地存储的前提下对接 API 做同步或备份。
- 详见原路线图中「数据与后端」思路；对「只卖账号 + 本地 App」而言非必须。

---

## 六、建议实施顺序

1. **阶段 1**：已完成（安全与配置）。  
2. **阶段 2**：在 [售卖账号模式说明](./售卖账号模式说明.md) 中选定**方式一**或**方式二**并落地（方式一无需改代码，方式二需加登录校验接口 + 前端调用）。  
3. **阶段 3**：按实际上架与合规需求做隐私政策、应用商店与更新策略。  
4. **阶段 4**：仅在有云端/同步需求时再做。

当前代码已支持「售卖账号 + 本地 App」：方式一可直接用；方式二需增加登录时调后端校验并保留现有本地逻辑。
