# 商用与交付

## 商业模式

当前项目更适合“售卖账号 + 本地 App”的交付方式，即用户通过安装包使用应用，业务数据主要保存在本机。

## 两种交付模式

### 方式一：一客户一包

特点：

- 无需后端
- 为每个客户单独构建安装包
- 可在构建时写入专属账号或品牌信息

适用场景：

- 客户数量可控
- 偏定制化交付
- 不依赖统一应用商店分发

### 方式二：通用包 + 多账号

特点：

- 只维护一个通用安装包
- 用户通过账号密码登录
- 需要一个最小登录校验服务

适用场景：

- 希望统一分发
- 需要后续增删账号、设置有效期
- 有一定运维能力

## 风险与控制

若担心 APK 转发或一号多用，可优先考虑方式二，并增加：

- 设备绑定
- 定期联网校验
- 账号状态管理

若使用方式一，则更偏向商务约束与人工交付管理，技术层面很难完全防止转发。

## 推荐实施顺序

1. 先确认采用方式一还是方式二
2. 方式一优先完善构建与交付流程
3. 方式二再补最小认证服务与账号管理
4. 最后补充隐私政策、发布规范与更新策略

## 商用注意事项

- 生产环境不要使用默认账号
- 品牌名称与登录凭据建议通过环境变量注入
- 若需要长期演进，尽量提前规划版本、签名、账号与更新策略
