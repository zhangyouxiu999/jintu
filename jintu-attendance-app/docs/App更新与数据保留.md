# 更新 App 不破坏现有数据

当前数据全部存在**本机**（localStorage，键前缀 `jintu_attendance_`）。要做到更新 App 后数据不丢，需满足下面几点。

---

## 一、覆盖安装时数据会保留（默认行为）

- **Android / iOS**：使用 Capacitor 时，数据在系统为 App 分配的应用数据目录里（WebView 的 localStorage 即存在此处）。  
- **覆盖安装**（同一包名 / Bundle ID 的更高版本覆盖旧版本）时，系统**不会**清空该目录，因此 **localStorage 会保留**，班级、学生、考勤等数据不会因更新而丢失。  
- **卸载后再安装**：应用数据目录会被系统删除，数据会丢失。所以要让用户「更新」而不是「卸载重装」。

**结论**：只要用户是**直接安装新版本 APK/IPA 覆盖旧版本**（或通过应用商店的「更新」按钮），且包名/App ID 未改，现有数据不会被破坏。

---

## 二、发布更新时必须遵守的约定

### 1. 不要改应用身份

- **Android**：`applicationId`（如 `io.jintu.attendance`）必须与已发布版本一致。  
- **iOS**：Bundle Identifier 必须一致。  
- 若修改了这些，系统会视为**另一个 App**，覆盖安装不会发生，用户相当于装了一个新 App，旧数据在旧 App 里，不会带到新 App。

### 2. 不要随意改存储键名或结构

- 数据键在 `src/store/storage.ts` 的 `STORAGE_KEYS` 和 `PREFIX`（`jintu_attendance_`）中定义。  
- **不要**在新版本里改名或删除已有键（如 `jintu_attendance_classes`、`jintu_attendance_students` 等），否则旧数据读不到，相当于数据「丢失」。  
- 若未来要调整数据结构（例如增加字段、换格式），应做**兼容读取 + 迁移**：新版本先按旧格式读，再在内存里转成新格式并写回新键或新结构；不要直接删旧键或改旧键含义。

### 3. 可选：为存储加版本号与迁移

- 在 localStorage 里存一个「数据版本」键，例如 `jintu_attendance_schema_version`，值为数字。  
- 每次发版若数据结构有变更，在 `db.ts` 的 `init()` 里：  
  - 读当前版本号；  
  - 若小于当前代码要求的版本，按顺序执行一段段迁移逻辑（从旧结构读到内存，转成新结构再写回）；  
  - 最后把版本号更新为当前版本。  
这样以后改结构时，老用户更新后会自动完成迁移，数据不丢且格式统一。

---

## 三、建议给用户的说明

在更新说明或设置页里可写一句：

- 「请通过**覆盖安装**或应用商店的**更新**升级，不要先卸载再安装，否则本机数据会被清空。」  
若日后做了「导出/备份」功能，可再加：「重要数据建议在设置中导出备份后再进行大版本更新。」

---

## 四、小结

| 做法 | 说明 |
|------|------|
| 覆盖安装、保持包名/App ID 不变 | 系统会保留应用数据目录，localStorage 自然保留，**更新不破坏数据**。 |
| 不删、不改已有存储键和前缀 | 老数据仍能被新版本正确读取。 |
| 未来若改数据结构 | 用「版本号 + 迁移逻辑」在 init 时把旧数据转成新格式，再写回。 |
| 提醒用户勿卸载重装 | 避免误操作导致数据被清空。 |

按上述约定发布更新，即可在**不破坏现有数据**的前提下持续迭代 App。
